
# 第1章　重新理解语音基础模型：从“语音 token”到“多时间尺度连续系统”

## 导言

过去两三年，语音基础模型的发展速度非常快。无论是学术界还是工业界，人们都在尝试构建能够“听懂、会说、可对话、可跨模态推理”的统一模型。表面上看，这一波进展似乎与大语言模型的发展路径高度相似：文本领域先出现了统一的预训练语言模型，随后这些模型在理解、生成、指令遵循、多轮对话上不断外延；语音领域也正在经历一个从单任务系统向统一基础模型迁移的过程。两篇 2025 年的综述都明确指出，spoken language processing 正从为自动语音识别、说话人识别、语音翻译、情感识别等任务分别训练专用模型，逐步转向使用 spoken language models 或 speech language models 作为更通用的基础系统；只是这条道路在语音领域起步更晚，路径也更复杂。[Arora et al., 2025; Cui et al., 2025]

然而，如果我们只把语音模型看作“文本大模型的语音版本”，那么很容易忽略一个更根本的问题：语音与文本在信息结构、时间组织方式、表示边界和生成要求上并不相同。文本天然是离散符号序列，词、子词、句子以及标点构成了较为稳定的边界；而语音首先是连续波形，其中语义、韵律、时长、音色、情绪、说话人特征乃至环境线索同时纠缠在一起。Arora 等人的综述强调，语音是“没有明显 tokenization 边界、也没有语言信息与其他声学因素清晰分离”的连续波形；因此，语音模型从一开始就不得不面对文本模型几乎不需要面对的问题：到底应该把什么压缩成 token，又应该把什么保留为连续动态？[Arora et al., 2025]

正因为如此，今天大量所谓“统一”的语音模型，虽然已经能够实现 speech-in / speech-out 或 speech-text 联合建模，但其底层骨架往往仍然沿袭文本语言模型的思路：先把语音离散化为一串 token，再用自回归语言模型对这些 token 做 next-token prediction，最后再通过声码器还原为波形。Cui 等人的综述把这种主流 SpeechLM 总结为三个部件：speech tokenizer、language model、token-to-speech synthesizer（vocoder）；从工程角度看，这一路线的优势非常明显：它允许研究者复用文本 LM 的训练和推理范式，也使 speech-text 联合建模变得更直接。[Cui et al., 2025]

但与此同时，这一路线的局限也越来越明显。第一，语音中的高层语义与低层声学细节往往难以被同一种 token 充分承载。第二，理解和生成虽然都处理“语音”，却并不要求模型在同一层次上关注信息：理解更依赖稳健的语义压缩和上下文整合，生成则更依赖细粒度的声学展开、韵律控制和说话风格保持。第三，语音交互天然包含时间连续性、多说话人对齐、抢话、中断、停顿和实时反应，这些都不是“把语音转成 token，再按文本方式逐个预测”可以轻松解决的问题。最近的模型与综述实际上已经在不断暴露这一点：语义 token 与声学 token 之间的取舍、层级式 token 生成、speech-text 的对齐方式、流式与全双工交互、连续特征与离散特征的折中，几乎都说明当前系统并未真正回答“语音基础模型的骨架到底应该是什么”这一问题。[Cui et al., 2025; Arora et al., 2025]

本章的目标，不是简单罗列近年语音模型的名称，而是借助最新综述建立一个更清晰的认识框架。我们会首先回顾语音模型为何从传统“ASR + LLM + TTS”级联系统走向 SpeechLM/SLM；随后讨论当前主流路线如何围绕 tokenization 展开，以及这种设计为什么既推动了统一模型的进展，又在更深层次上限制了语音建模；接着分析语音理解与语音生成之间并不对称的建模需求；最后，在这些矛盾自然累积的基础上，引出一种更适合语音的系统观：语音基础模型或许不应只是“文本模型外面套一层 speech token”，而应当是一个按多时间尺度组织的连续动力系统，在不同频率上分别承担声学细节、局部内容、长程语境与交互意图的建模。

换句话说，本章会给出一个核心判断：今天主流的语音语言模型已经把问题推进到了“统一接口”的阶段，但真正重要的下一步，可能不是再发明一种更好的 token，也不是在现有框架上增加一个局部技巧，而是重新思考语音模型的统一骨架。本书后续章节将围绕这一判断展开。

## 1.1　从“ASR + LLM + TTS”到 SpeechLM：统一语音建模的第一步

如果从系统形态看，最直接的“语音大模型”方案其实并不复杂：先用自动语音识别（ASR）把用户说的话转成文本，再把文本送入大语言模型生成回答，最后用语音合成（TTS）把回答读出来。这种级联系统在工程上非常自然，也曾长期被视为语音交互的主流实现方式。Cui 等人的综述就把它概括为 “ASR + LLM + TTS” 框架，并指出这是把文本大模型快速扩展到语音交互场景的最直接方案。[Cui et al., 2025]

但是，正如该综述所强调的，级联系统存在三个非常关键的问题。其一是信息损失。语音并不只携带“说了什么”的语义信息，还包含音高、音色、说话速度、重音、停连、情绪和风格等副语言信息；当系统把语音先转成文本时，这些信息在进入中间的文本 LLM 之前就已经被大幅压缩甚至完全丢失。其二是时延。ASR、LLM、TTS 三个模块级联之后，任何一个环节的复杂度都会被放大到系统级交互时延上，尤其在实时对话场景中更为明显。其三是误差累积。前一级的识别错误会被传递到后续的理解和生成阶段，最后在 TTS 输出中固化下来。Cui 等人因此把 SpeechLM 的兴起解释为对上述三个问题的回应：如果模型能够直接对语音进行端到端建模，并在模型内部同时处理理解和生成，理论上就能减轻信息损失、降低级联时延，并缓解误差逐级扩散的问题。[Cui et al., 2025]

Arora 等人的综述从更宽的视角给出了这一转变的理论化描述。他们指出，今天文献中被称为 spoken language models 的模型并不只有一种，而至少可以分为三类。第一类是 pure speech LM，即直接建模语音分布 p(speech) 的模型，典型地使用无标签语音数据和 next-token prediction 目标训练；第二类是 speech+text LM，即联合建模文本和语音分布 p(text, speech) 的模型；第三类是 speech-aware text LM，它们通常从已经具备强指令能力的文本 LLM 出发，再接入预训练语音编码器，用以学习条件分布 p(text|speech, text)，在保留文本大模型能力的同时获得对语音输入的理解能力。[Arora et al., 2025]

这一分类很重要，因为它说明“统一语音模型”并不是单一路线，而是一个由不同目标函数、不同输入输出形式和不同初始化策略构成的谱系。pure speech LM 更像文本领域的 GPT 式预训练语言模型，是对“语音作为序列”这一问题的直接回答；speech+text LM 试图同时处理 speech 和 text 两种序列，因此更接近跨模态联合建模；speech-aware text LM 则更接近“在文本大模型基础上接入语音理解前端”的路径，其优势是能较强地继承文本 LLM 的指令遵循与推理能力，但其输出往往仍偏向文本。[Arora et al., 2025]

从历史上看，这三类模型恰好对应了语音基础模型发展的三个阶段。最早的一批 pure speech LM，例如 GSLM，尝试直接在离散化后的语音 token 上进行语言建模。这些模型的重要性在于，它们第一次比较系统地提出：语音也可以像文本一样被当作序列来预训练，并在一定程度上表现出“语言模型式”的泛化。随后，随着 AudioLM、Moshi、Mini-Omni、SpeechGPT、SpiRit-LM 等工作出现，研究者开始探索 speech+text 联合建模、层级式 token 生成、文本与语音交错序列建模以及实时对话等更复杂的问题。与此同时，以 SALMONN、Qwen-Audio-Chat 为代表的 speech-aware text LM 说明：文本大模型并不一定要被完全改造成 speech-in / speech-out 系统，它们也可以先在 speech-in / text-out 的设定下发展出广泛的语音理解能力。[Arora et al., 2025]

于是，我们就能看清一个非常关键的事实：SpeechLM/SLM 的出现，并不是简单用一个“更大”的模型替代旧流水线，而是把原本分裂的三种能力——语音表征、语言建模、语音生成——重新组织到了一个更紧密耦合的系统里。Cui 等人因此把 SpeechLM 明确界定为能够进行 end-to-end speech interaction 的自回归基础模型；这一定义把“能接收并生成语音”视为 SpeechLM 与传统 TextLM 的根本区别。[Cui et al., 2025]

不过，也正是在这里，第一个更深层的问题开始浮现：当 SpeechLM 把语音表征、语言建模和语音生成整合成一个系统之后，这个系统内部仍然往往延续着“tokenize → autoregress → decode”的文本式范式。换言之，SpeechLM 解决了“模块级联”的问题，却没有彻底回答“语音该如何被表示和处理”的问题。它把旧系统的三段式外部流程，变成了模型内部的三段式结构；模型变得更统一了，但统一的中心仍然是 token 序列。这个问题将在后续几节不断显现出来。

## 1.2　为什么语音不能被简单看成“另一种文本”

在文本建模中，tokenization 通常不是一个哲学问题，而更多是工程问题。词、子词、字符、BPE 单元虽然不同，但无论如何，文本总是由离散的书写符号构成。对文本来说，tokenization 更多是在不同离散粒度之间做权衡：是偏向词级、子词级还是字符级，是选择更紧凑的词表还是更开放的组合能力。这些选择会影响长度、词表规模和泛化，但很少会动摇“文本是离散序列”的基本前提。

语音不是这样。Arora 等人的综述用一句非常重要的话概括了这一点：与文本不同，语音是连续波形，它既没有显然的 tokenization 边界，也不存在语言信息与其他声学因素之间天然的清晰分离。[Arora et al., 2025] 这意味着，语音模型在开始建模之前就要先回答一个文本模型几乎不必回答的问题：究竟应该把连续波形压缩成什么样的表示，才既不丢掉任务所需的信息，又能让后续模型有效学习？

这件事之所以困难，是因为语音中至少同时存在几类性质截然不同的信息。第一类是语义和语言内容，也就是“说了什么”；第二类是语音组织相关的信息，例如音素、音节、发音方式、时长和局部节奏；第三类是副语言与说话人相关的信息，例如音高、音色、情绪、风格、口音和身份；第四类则是场景和交互信息，例如停顿、重叠、接话意图、环境噪声和谈话状态。这些信息不是分层存放在语音中的，而是共同存在于同一条时间轴上，并且以不同时间尺度耦合在一起。

因此，当一个模型试图把语音压缩成 token 时，它事实上是在做多重投影：它必须决定哪些信息保留为稳定、高层、可概括的模式，哪些信息被压缩到更细粒度的表示里，哪些信息被丢弃，哪些信息留给后续声码器或解码器去恢复。Cui 等人的综述把 speech tokenizer 分成三类：语义理解导向、声学生成导向以及混合目标导向。这个分类本身就证明，所谓 speech tokenizer 并不是单纯的输入格式转换器，而是模型对“语音本质上是什么”这一问题的先验选择。[Cui et al., 2025]

如果 tokenizer 以语义理解为目标，它就倾向于把语音压缩为对内容最敏感、对具体声学细节相对不敏感的表示。这类 tokenizer 往往有利于 ASR、语义理解、语音翻译等任务，因为它们提炼的是“说了什么”。如果 tokenizer 以声学生成为目标，它就会尽可能保留可重建高保真波形所需的细节，例如局部频谱结构和细粒度声学线索。这类 tokenizer 更适合 TTS、语音延续、说话人风格保持等任务，因为它们保留的是“怎么说”。如果 tokenizer 试图兼顾两者，它就必须在语义压缩与声学保真之间寻找平衡，往往需要额外的结构设计或层级式表示。[Cui et al., 2025]

这说明语音建模与文本建模之间最大的差别之一，不在于“输入模态不同”，而在于“表示任务不同”。文本 token 本身大体就是语言学上的基本输入单位；语音 token 则是一个模型设计者人为构造出来的中间表示，它不是语音的自然边界，而是某种任务假设下的最佳压缩结果。语音模型之所以比文本模型更难统一，本质原因也在这里：文本的输入结构在进入模型之前已经相对清楚，而语音的输入结构恰恰需要模型体系本身去决定。

进一步说，文本理解和文本生成通常共享相同的符号空间：输入词与输出词处在同一种离散层次上，即使目标不同，模型看到和生成的仍然是同类单位。语音则往往不是这样。对于理解任务，模型可能更想看到稳定的语义单元；对于生成任务，模型又需要输出足够细致的声学单元。这意味着语音系统未必适合用一套完全相同的表示来同时服务输入和输出，更不一定适合用同一种时间粒度对所有问题建模。这种输入/输出表示的不对称性，正是后续很多统一语音模型反复撞上的墙。

所以，语音不能被简单看成“另一种文本”，并不只是因为它是音频，而是因为它迫使模型面对一个文本领域被大大简化了的难题：如何在连续信号中同时组织语义、结构、风格和交互。如果忽略这一点，我们就会不断把语音问题错误地转写为“如何给语音找到更合适的 token”，而看不到更大的系统问题：也许语音的核心并不是 token 本身，而是不同信息层级在不同时间尺度上的动态耦合。

## 1.3　当前统一语音模型的主流技术路线：tokenizer、序列模型与 vocoder

尽管语音与文本存在根本差异，但当前统一语音模型最成功的一条路线，仍然是把语音组织成可以被语言模型消费的序列。Cui 等人的综述对这一范式做了非常清晰的拆解：SpeechLM 由三个核心组件构成，即 speech tokenizer、language model 和 vocoder。tokenizer 把波形转成离散或连续表示，language model 在这些表示上做序列预测，vocoder 再把预测结果还原成语音。这个三段式架构之所以成为主流，并不是因为它是语音唯一可能的结构，而是因为它在当下最容易继承文本大模型的经验和工具链。[Cui et al., 2025]

### 1.3.1　speech tokenizer：先决定“什么值得被压缩”

speech tokenizer 是整个 SpeechLM 的入口，它也是现阶段语音模型中最关键、争议最大的部件。Cui 等人的综述指出，speech tokenizer 可以输出离散 token，也可以输出连续 embedding；两者都能作为后续语言模型的输入，其目标是在降低维度的同时保留建模所需的关键语音信息。[Cui et al., 2025]

按照优化目标的不同，speech tokenizer 可以分为三类。

第一类是**语义理解导向 tokenizer**。这类方法把重点放在“把语音中的含义提取出来”，因此往往使用自监督语音表示模型加量化器的结构，把波形映射为更接近语义内容的 token。HuBERT、wav2vec 2.0、w2v-BERT、USM、WavLM 等工作都可以被放在这一类谱系中。Cui 等人的综述指出，这类 token 有利于 ASR 等以内容为中心的任务，因为它们更准确地捕捉了 speech meaning。[Cui et al., 2025]

第二类是**声学生成导向 tokenizer**。这类方法更关心高保真语音重建，因此往往采用 encoder-quantizer-decoder 的 neural codec 结构，让 token 尽可能保留局部频谱与声学细节，用于高质量语音合成或重建。SoundStream、EnCodec 及其后续 codec 类模型就是这一思路的代表。此类 token 的优点是声学信息丰富，适合生成；缺点是它们未必是最好的语义单元。[Cui et al., 2025]

第三类是**混合目标 tokenizer**。这类方法试图同时兼顾语义与声学，通常做法不是发明完全不同的新结构，而是在 acoustic tokenizer 上加入语义蒸馏或混合约束。例如 SpeechTokenizer、Mimi 等工作都体现了这一方向：既希望 token 可被语言模型建模，又希望它们在解码时不至于丢掉太多可感知细节。[Cui et al., 2025]

从系统角度看，这三类 tokenizer 的存在本身就说明：所谓统一语音模型的“统一”，首先并不统一在输入层，而统一在后面的序列建模范式上。tokenizer 的任务不是简单把语音切段，而是在理解优先、生成优先与两者兼顾之间做不可避免的取舍。

### 1.3.2　sequence model：借用文本 LM，但问题并没有消失

在 tokenizer 把语音压缩成序列之后，当前主流 SpeechLM 往往直接沿用文本 LM 的建模方式。Cui 等人的综述明确指出，SpeechLM 通常采用 transformer 或 decoder-only 结构来进行自回归语音生成；如果输入是离散 speech token，那么只需要把文本词表和 embedding 替换成 speech 词表和 embedding，甚至可以把文本词表与语音词表扩展为同一个联合词表，从而让同一个语言模型处理 speech 与 text 两种模态。[Cui et al., 2025]

这条路线的好处很直接。第一，它允许研究者最大程度复用文本大模型的训练经验。第二，它使得 speech-text 联合建模、指令微调和后对齐在工程上都更顺畅。第三，它让语音模型能够自然地继承文本领域已经被验证过的自回归建模、指令调优、偏好优化乃至工具调用思路。Arora 等人的综述也强调，很多 speech-aware text LM 正是通过把预训练语音编码器接到文本 LLM 上，从而保留文本模型的 instruction-following 能力，并获得语音推理能力。[Arora et al., 2025]

但是，sequence model 虽然借用了文本 LM 的形式，语音的困难却并没有因此消失。Arora 等人的综述指出，speech sequence 与 text sequence 在一个重要方面不同：音频 token 往往天然包含粗细不同、粒度不同的层次。例如 phonetic token 可以视为较粗的内容单元，audio codec token 则更像细粒度的声学单元；因此，很多语音模型不得不采用**层级式 token 生成**策略，而不是像文本那样只预测单一层次的 token。[Arora et al., 2025]

他们把常见的层级生成总结为几种形式：先生成 coarse token 再生成 fine token；把粗细 token 交错建模；先做 temporal generation 再做 depth generation；或使用 delay pattern 等方式组织多路输出。[Arora et al., 2025] 这一点非常关键，因为它说明当语音模型表面上看似仍在做“next-token prediction”时，实际上已经开始偏离文本语言模型最简单的序列假设。模型之所以要设计多个 token 通道、对齐多个时间粒度，本质上是因为语音中的内容层和声学层并不服从同一种离散序列组织方式。

### 1.3.3　vocoder：统一模型中最容易被忽视的“生成下半身”

在很多关于语音基础模型的讨论中，vocoder 常被当作“最后再接一个解码器”的附属部件，但它实际上决定了系统是否能够把模型内部表示真正转化为自然、低延迟、可交互的语音输出。Cui 等人的综述提醒我们，vocoder 本身也有多种路线：自回归、GAN、flow-based、VAE-based、diffusion-based 等，各自对应不同的保真度—时延权衡。[Cui et al., 2025]

从实时交互角度看，这一点尤其重要。该综述在讨论实时交互评测时指出，GAN-based vocoder 往往因为效率高而在实际交互系统中更受欢迎，而 diffusion-based vocoder 虽然在高保真上更强，却更容易带来时延负担。[Cui et al., 2025] 换言之，今天很多“统一语音模型”的瓶颈并不完全在语言模型本身，而在于其整体系统必须同时满足理解、生成、自然度与实时性要求。在一个真正面向语音交互的系统里，vocoder 并不是后处理细节，而是决定生成路径能否落地的重要组成。

### 1.3.4　从 turn-based 到 full duplex：语音不是一问一答的静态序列

当前语音模型的另一个关键演化方向，是从“完整听完再完整回答”的传统交互范式，转向流式甚至全双工对话。Cui 等人的综述指出，早期多数 SpeechLM 仍然遵循传统 speech interaction paradigm：先接收一整段输入，再输出一整段回应。但这种模式与真实人类对话并不相符，因为自然对话中存在抢话、停顿、重叠、边听边说和被打断后重新组织输出等复杂现象。[Cui et al., 2025]

因此，实时语音模型需要的不只是更快的推理，而是对交互结构本身的建模。综述把这一演化分成几个层次：最初是采用流式 tokenizer 和流式 vocoder，让系统无需等待整段编码完成；再往前是 full-duplex modeling，使模型能够在两个通道上同时处理输入与输出，并支持用户中断与同步响应。Parrot、Moshi、LSLM 等模型都可以被放在这一谱系中。[Cui et al., 2025]

这一变化说明了什么？说明语音统一模型并不是把“输入模态”从 text 换成 speech 那么简单。只要系统真正进入对话场景，它就不得不面对多通道时间对齐、说话人切换和交互节奏的问题。这些问题在本质上都和语音的连续时间性有关，而不是“token 序列长度变长”这么简单。也正因此，语音基础模型未来的骨架很可能不仅要统一表示，还要统一时间组织。

## 1.4　token 化范式的成就与代价：语义 token、声学 token、混合 token 与连续表示

从今天的研究进展看，token 化路线无疑是成功的。它让语音建模第一次真正与文本语言模型范式接轨，也让 speech-text 联合训练、指令微调和跨模态迁移变得可操作。没有 token 化，就不会有今天这一波 SpeechLM/SLM 的快速扩张。但正因为它过于成功，研究者也越来越容易把“找到更好的 token”当成语音统一模型的核心问题。事实上，综述和新近论文恰恰提醒我们：token 化既推动了统一模型，也制造了新的结构性张力。

### 1.4.1　为什么离散 token 仍然占主流

Arora 等人的综述指出，在目前的 pure speech LM 和 speech+text LM 中，离散 speech token 仍然是最常见的建模单位，而 speech-aware text LM 则更常采用连续表示作为输入（当然也有例外）。原因很简单：离散 token 最容易与语言模型的 next-token prediction 目标兼容；一旦语音被转写为 token ID 序列，模型就可以像处理文本一样处理语音。[Arora et al., 2025]

Cui 等人的综述也强调，大多数 SpeechLM 使用离散 token 的重要原因就在于：它们的建模过程与 TextLM 中的 text token 几乎同构。[Cui et al., 2025] 这给研究者带来了巨大的工程红利。离散 token 允许使用现成的 decoder-only transformer，允许直接扩展词表做 speech-text joint vocabulary，允许借助文本预训练 checkpoint 做 warm start，也允许把语音数据纳入与文本数据相似的序列建模管线中。

因此，我们不应轻易否定 token 化路线。它不是错误的，而是在现阶段最有效的折中方案。问题在于，当我们把它当作最终答案时，就会忽视它背后的代价。

### 1.4.2　语义 token 的优势：更容易理解、更容易对齐文本

语义 token 的最大优势，是它们与“内容”关系更紧密。Cui 等人指出，大多数 SpeechLM 在实践中都偏好 semantic token，因为它们能够捕捉 crucial contextual information，并在 ASR、语义理解、语音翻译等任务上表现更好。[Cui et al., 2025] Arora 等人的综述则进一步指出，phonetic token 往往具有较强的语言学可解释性，适合作为 speech sequence 的粗粒度单元。[Arora et al., 2025]

从跨模态角度看，语义 token 还有另一个重要优点：它们更容易与文本 LLM 对齐。因为文本本质上只显式编码语义与语言内容，越是接近“内容”的 speech 表示，越容易借助文本监督、文本 embedding 或文本行为对齐来进行迁移。这也是很多 speech-aware text LM 能够快速获得能力的原因：它们不必完整建模波形，而只需学会把 speech 映射到文本语义空间附近即可。[Arora et al., 2025]

### 1.4.3　语义 token 的代价：声学细节被压扁了

但语义 token 的问题同样明显。Cui 等人的综述非常直接地指出：虽然 semantic token 可以生成在语义上连贯的语音，但往往缺乏 prosody、timbre 等 expressive elements；要恢复这些高频细节，常常还需要后处理，例如 diffusion model，而这又显著增加了时延。[Cui et al., 2025]

换句话说，语义 token 的优势恰恰来自于它们压缩掉了大量细节；而这些被压缩掉的细节，正是自然语音生成最依赖的部分。当任务是“听懂”时，这种压缩是好事；但当任务变成“说得像人”，这种压缩就成了限制。这说明，语音中的“好表示”并不是单一的：对于理解来说，过于细的表示可能是噪声；对于生成来说，过于粗的表示又会让输出失去生命力。

### 1.4.4　声学 token 的优势：保真、自然、可还原

与之相对，声学 token——尤其来自 neural codec 的 token——对语音合成与重建更友好。Cui 等人指出，acoustic token 旨在保留高保真语音重建所需的声学特征，因此更适合高质量语音生成。[Cui et al., 2025] Arora 等人的综述也把 audio codec token 视为 finer token，并指出很多层级式语音模型都把它们作为细粒度输出层来处理。[Arora et al., 2025]

这类 token 对“怎么说”更敏感：说话人音色是否稳定、停顿是否自然、韵律是否流畅、情绪是否一致、背景声场是否协调，都更容易通过 rich acoustic representation 来建模。对于真正的 speech-in / speech-out 模型来说，如果没有某种形式的 acoustic-level 表示，模型很难生成足够自然的输出。

### 1.4.5　声学 token 的代价：内容建模压力陡增

但 acoustic token 的不足也同样清楚。Cui 等人的综述指出，虽然 acoustic token 有利于高保真音频生成，但它们在内容生成上常常会出现 inaccuracies，换言之，模型更容易“说得像”，却不一定“说得对”。[Cui et al., 2025] 这是因为 acoustically rich token 同时携带大量与语义无关或弱相关的细节，语言模型必须在这些复杂局部模式中再去学习长程内容依赖，其难度显著高于在相对抽象的 semantic token 上学习。

这一点非常像在视觉建模中直接对像素做序列建模：并非不能做，而是模型很容易把大量容量消耗在低层统计结构上，而不易稳定抽取高层语义。语音中的 acoustic token 正存在类似问题。它们不是“差”的表示，而是“更偏向生成”的表示。把这类表示强行同时作为理解和生成的统一核心，往往会让模型在两边都付出代价。

### 1.4.6　混合 token 与层级 token：折中不是没有成本

正因为 semantic token 与 acoustic token 各有利弊，研究者自然会尝试混合二者。Cui 等人的综述总结了两种主要策略：一种是像 AudioLM 那样，把 semantic token 与 acoustic token 组织成层级式生成流程，先生成 coarse semantic token，再预测 finer acoustic token；另一种是用 mixed token 或 multi-stream token 直接联合建模语义与声学信息，在 Moshi、SpeechGPT-Gen 等模型中都能看到这种思想。[Cui et al., 2025]

Arora 等人的综述也系统讨论了 hierarchical token generation，指出常见策略包括 coarse-first-then-fine、interleaved coarse/fine tokens、temporal generation plus depth generation，以及 delay pattern 等。[Arora et al., 2025] 这些设计都说明，社区已经意识到“一种 token 解决全部问题”很难成立。

然而，折中并不意味着没有代价。Cui 等人明确指出，把 semantic 与 acoustic token 组合进单一序列，往往会拉长序列长度、提高建模复杂度并增加时延。[Cui et al., 2025] 这说明 token 路线虽然可以通过“多种 token”来修补单一 token 的不足，但代价是系统会越来越复杂：更多序列、更复杂对齐、更高时延、更困难的训练稳定性，以及更不清楚的误差归因。

### 1.4.7　连续表示：一种必要但尚未成熟的替代方向

除了离散 token，还有一条重要但尚未成为主流的路线，即直接使用连续 speech representation。Arora 等人的综述指出，continuous features 可以来自 mel-spectrogram，也可以来自自监督或监督预训练语音编码器的隐层表示；它们能够保留离散化容易丢失的细粒度信息，但需要修改传统 text-based pipeline，且存储和计算代价更高。[Arora et al., 2025] Cui 等人的综述同样指出，continuous features 是 speech tokenizer 的一个合法输出形式，并能够直接送入后续语言模型。[Cui et al., 2025]

这条路线之所以值得重视，不仅因为它可能保留更多语音细节，更因为它提醒我们：把语音先离散化再语言建模，并不是逻辑必然，而只是当前主流 engineering choice。语音本来就是连续信号，如果模型的骨架能够更自然地处理多时间尺度连续状态，那么离散 token 也许不必承担今天这么多结构性责任。

到这里，我们就能看到 token 化范式的全貌：它是统一语音模型的发动机，同时也是其瓶颈。它把复杂的连续语音问题翻译成了“序列建模问题”，从而大幅推动了进展；但它也把语音内部本来分属不同时间尺度、不同功能层次的信息，压到单一路径或少数几条路径上处理，于是不可避免地造成理解与生成、语义与声学、效率与保真之间的长期拉扯。

## 1.5　统一模型中的核心张力：语音理解与语音生成并不是同一个任务的镜像

如果说上一节讨论的是“用什么表示语音”，那么本节讨论的就是另一个更深的问题：语音理解与语音生成是否真的应该被看成同一种建模任务的两个方向？表面上看，两者确实都是对 speech signal 的处理：理解是从语音到含义，生成是从含义到语音；一个是压缩，一个是展开。因此，人们很容易认为：只要找到一种足够好的统一 token，再加一个足够强的序列模型，理解与生成就可以在同一个框架里同时做好。

但越来越多的工作表明，这个假设并不稳固。DualSpeechLM 的出发点几乎可以看作对这一假设的直接反驳。该文摘要明确指出，统一 speech understanding 和 generation 面临两个核心挑战：第一，speech token 与 text token 之间存在巨大的 modality gap，因此把 text LLM 扩展成 speech LLM 往往依赖大量 paired data；第二，更关键的是，generation 和 understanding 偏好的信息层次不同——生成更受益于细致的 acoustic feature，而理解更偏好 high-level semantics。正因为这种差异，在单一统一模型中同时优化两类任务会非常困难。[Wang et al., 2025]

这段判断非常重要，因为它把问题从“模型还不够大”转向了“任务对信息层的需求根本不同”。也就是说，统一模型之所以难，不只是因为数据、参数或训练资源不足，而是因为理解与生成要求模型在不同层次上组织 speech information。

从理解的角度看，理想模型要做的是**稳定压缩**。它需要在说话速度变化、发音细节扰动、环境噪声、说话人口音变化甚至情绪变化之下，尽量抽取不变的语义和任务相关信息。对理解而言，很多低层声学波动本质上是 nuisance variables；模型需要学会对它们不敏感。正因如此，semantic token、phonetic token、speech-to-text alignment、explicit/implicit modality alignment 等方法会特别重要。[Cui et al., 2025; Arora et al., 2025]

从生成的角度看，理想模型做的却是**细致展开**。同一句文本，不同说话人、不同情绪、不同语速和不同语境会对应大量不同但都合理的语音 realizations。生成模型不仅要知道“说什么”，还要知道“怎么说、以什么风格说、在这一轮对话里为什么要这样说”。这里，声学细节不是噪声，而是输出质量的主体。因此 acoustic token、codec representation、prosody token、style token、speaker embedding、low-latency vocoder 等部件会格外重要。[Cui et al., 2025]

于是就出现了一个经常被忽视的事实：**理解要求模型对变化不敏感，生成要求模型对变化敏感**。理解追求不变性，生成追求可控变化；理解偏好抽象，生成偏好具象；理解偏好压缩，生成偏好展开。两者共享同一“语音”对象，但对语音中什么是信号、什么是噪声的判断恰恰相反。

DualSpeechLM 正是在这一点上提出了非常具有代表性的修正。它没有继续坚持“同一种 speech token 同时做输入和输出”的对称设定，而是采用 dual-token modeling：输入侧用更偏理解的 USToken，输出侧则交给 Acoustic GPT 生成 acoustic token，并在 Text LLM 内部把两条路径联合起来。[Wang et al., 2025] 该文进一步指出，这种设计的核心不是简单增加一个额外模块，而是承认 understanding 与 generation 需要不同层次的信息。其 Figure 3 也明确把 baseline 管线与 DualSpeechLM 做了对比：baseline 把 LLM 输入/输出都视为同一种 token，而 DualSpeechLM 则通过不同 token 路径分别处理理解与生成。[Wang et al., 2025]

换句话说，DualSpeechLM 提供了一个很有代表性的信号：统一模型未必要在“同一 token、同一通道、同一时间粒度”上统一。相反，更现实的统一可能恰恰来自**结构上的非对称性**——让理解与生成共享一部分高层表示与上下文能力，但在信息粒度、状态更新和输出路径上保持差异。

这也帮助我们重新理解许多近年来看似分散的设计。为什么有的模型喜欢 interleaved speech-text sequence？因为它们希望借助文本这一低熵、强语义模态来稳定 speech semantics。为什么有的模型要额外建 acoustic stream、prosody stream、style stream？因为生成路径不能只靠内容 token。为什么很多模型在真正对话时强调 streaming、full duplex、turn-taking 和 interruption handling？因为“生成语音”并不只是读出一句内容，而是要进入时间连续的交互过程。[Cui et al., 2025; Arora et al., 2025]

因此，本节最重要的结论是：统一语音模型的难点，并不只是把更多任务塞进一个大模型里，而在于理解和生成从一开始就不是镜像关系。它们共享知识，但不共享全部建模需求；共享高层意图，但不共享全部状态变量；共享语境，但不共享全部时间尺度。任何真正面向语音的统一骨架，都必须正面处理这种不对称性。

## 1.6　从“统一接口”到“统一骨架”：现有路线真正缺少的是什么

如果把最近几年的语音模型放在一起看，会发现一个很有意思的现象：社区已经非常会做“统一接口”了，但还不太会做“统一骨架”。所谓统一接口，是指模型能接受 speech、text 或两者混合作为输入，并在某种设定下输出 text、speech 或两者兼有的结果；从产品或 benchmark 的角度看，这当然很重要，因为它决定了模型能不能覆盖多任务、多模态和多轮交互。今天的许多 SpeechLM/SLM 已经在这方面取得了非常显著的进展。[Cui et al., 2025; Arora et al., 2025]

但如果进一步追问：这些模型内部究竟以什么方式统一了语音？答案往往并不令人满意。大多数系统仍然沿袭“先抽特征/量化，再送入序列模型，再经解码器还原”的文本式组织逻辑。哪怕模型已经引入 mixed token、hierarchical token、speech-text interleaving、多个输出通道、流式 encoder/vocoder，本质上也仍然是在文本 LM 的骨架上为 speech 额外开辟输入输出接口。Cui 等人的综述在形式化语言模型部分几乎是直接说明了这一点：SpeechLM 往往就是把 text embedding 替换或扩展为 speech embedding，从而让标准 language model 同时消费 speech token 与 text token。[Cui et al., 2025]

这不是说这种做法不对，而是说它很可能只是**过渡性的正确**。因为从系统论的角度看，语音中最难的部分并不在“如何接进模型”，而在“模型内部如何按不同时间尺度组织信息”。如果内部状态本身仍然只围绕一个或几个 token 序列展开，那么模型其实只是把语音外部地编码成文本式对象，而没有真正内生地表达语音的连续、多频率和多层次结构。

两篇综述在“未来方向”部分都间接指向了这一点。Cui 等人指出，当前 SpeechLM 对不同 component choice 的理解仍不充分，三大组件之间的对比研究不足；他们还特别提出 end-to-end training 是一个重要开放问题，即是否能够把梯度从 vocoder 的输出一路反传回 tokenizer 的输入，使整个模型真正作为一个整体来优化。[Cui et al., 2025] 这实际上是在问：今天三段式内部结构是不是仍然过于松散？

Arora 等人的综述则从另一个角度指出，当前 SLM 的开放问题包括：speech 与 text 最佳结合方式仍不清楚，不同 modality adapter 和 interleaving 方式缺乏系统比较；更重要的是，当前 SLM 往往又大又慢，不适合 real-time 或 on-device 使用，而其中一个内在原因恰恰是系统组合了多个预训练组件，这些组件经常拥有不同架构和不同 frame rate。[Arora et al., 2025] 这段观察尤其值得注意，因为它触及了一个容易被忽视的事实：现有很多“统一模型”之所以笨重，不只是因为参数多，而是因为它们在本体上是拼接起来的。speech encoder、modality adapter、text LLM、speech decoder/vocoder 常常来自不同预训练体系，各自工作在不同时间分辨率和不同状态空间中，统一只发生在接口层，而没有发生在动态结构层。

从这个意义上说，今天的统一语音模型更像“联邦系统”，而不是“单一系统”。它们能一起工作，但未必共享同一种内在时钟；它们能互相传递表示，但未必共享同一种状态组织原则。于是，很多现实问题就随之出现：为什么对齐那么难？因为不同模块处在不同表征空间。为什么时延很难降下来？因为多个模块在不同 frame rate 上串联。为什么理解和生成难以同时优化？因为共享的只是一条或几条 token 路径，而不是一套真正分层分频的统一状态系统。

也正因如此，如果我们想真正从综述中走出下一步，就必须把问题从“接口集成”提升到“骨架设计”。换句话说，下一代语音基础模型真正需要回答的，不是“能不能同时做 ASR、TTS、S2ST、对话与音频理解”，而是“模型内部应当如何组织不同频率、不同层次、不同功能的状态，使理解和生成能够在同一系统中各得其所”。

这里，“统一骨架”至少意味着四件事。

第一，它要能够容纳**不同时间尺度**。局部声学细节变化极快，韵律与短时模式变化较快，词与短语级内容变化较慢，长程语境、说话风格与交互意图变化更慢。把这些都压到同一个离散步长上处理，既低效，也不自然。

第二，它要能够容纳**不同方向的信息流**。理解更像自下而上的压缩：从局部连续信号逐步抽象出稳定语义。生成更像自上而下的展开：从高层意图与语境逐步落实到具体声学波形。一个真正统一的语音骨架，不能只是一条单向 next-token chain，而应当允许不同层级间存在不同的更新频率与不同方向的条件依赖。

第三，它要能够容纳**不同模态在不同频率上的作用差异**。文本在语音系统中未必需要与 speech 对等处理。对很多任务而言，文本更像低频语义锚点和外部约束；speech 则承担高频声学展开和交互时序。把 text token 和 speech token 简单拼进同一词表，虽然方便，却未必符合两种模态在系统中的真实角色。

第四，它要允许**端到端优化与局部专门化同时存在**。语音理解和生成不应被完全拆成独立模块，但也不必被迫共享全部低层细节。换言之，系统既要统一，又要保留结构性分工。

于是，本章前面关于 token 化张力、理解/生成不对称性、streaming 与 duplex 需求、speech-text 对齐困难的讨论，就在这里汇合到一个更大的问题上：也许语音基础模型真正需要的，不是“更像文本”，而是“更像语音”。而“更像语音”的核心，不是换一个更聪明的 tokenizer，而是承认语音本身就是一个跨时间尺度的连续系统。

到这里，我们就可以从综述自然过渡到本书将强调的前瞻性观点：未来的语音基础模型，很可能需要从“token 序列上的统一建模”转向“多时间尺度连续状态上的统一建模”。下一部分的展望，将在这一点上进一步展开。

# 展望：从 token 统一走向多时间尺度多模态连续语音基础模型

如果把前面几节的讨论浓缩成一句话，那就是：现有统一语音模型已经证明了 speech 不能再只是 ASR、NLP、TTS 三个模块的松散串联，但它们还没有真正找到最适合语音的统一骨架。今天的主流解法，实质上是在文本 LM 外面加上 speech tokenizer 和 speech decoder，把连续语音压缩成可被文本式语言模型处理的对象。这种做法在当下非常有效，也很可能仍会继续产出强模型；但从长远看，它更像一个过渡范式，而不是终局范式。

本书想进一步推进的问题是：**如果我们不是从“如何把语音变成 token”出发，而是从“语音本身是一种怎样的时间系统”出发，那么语音基础模型应该如何搭建？**

## 一、语音模型的根本单位也许不是 token，而是不同频率上的状态

文本模型之所以天然适合 token 序列，是因为文本的组织本来就相对离散。而语音不同。语音首先是一条连续信号，它在极短时间尺度上表现为波形与谱结构，在稍长时间尺度上表现为发音、音节、停连、重音和局部韵律，在更长时间尺度上表现为词语义、句法组织、话轮推进、说话人风格以及交互意图。这些信息不是层层切开排好的，而是同时叠加在同一条时间轴上。

因此，我们不妨提出一个更适合语音的起点：语音模型的基本对象不应只是“某种离散 token 序列”，而应当是一组**在不同时间尺度上演化的连续状态**。其中，最快的状态负责局部声学细节与发音动力学；中等速度的状态负责音节、词和短时内容组织；更慢的状态负责说话人风格、长期语境、对话策略和交互意图。模型不是先把所有信息压成同一种单位再统一处理，而是让不同频率的状态各自承担不同的信息职能，并通过跨层耦合完成理解与生成。

这种视角的优点在于，它更接近我们从综述中看到的所有“补丁性设计”背后的共同动机。层级 token 生成，其实是在用多种离散粒度近似不同频率的信息；semantic/acoustic dual token，其实是在承认理解与生成分别依赖较慢和较快的信息；streaming 和 full duplex，其实是在承认语音交互不是一条静态序列而是持续演化的时间系统。也就是说，社区已经在不断用工程办法触碰多时间尺度问题，只是还没有把它提升为骨架原则。

## 二、理解与生成不应再被设计成完全对称的两面

如果接受“语音是多时间尺度系统”的出发点，那么统一模型里的理解与生成就不应被设计为同一表示空间里的对称操作。理解应当是一个自下而上的过程：高频状态先对局部连续声学做感知和滤波，中频状态把这些局部动态压缩成相对稳定的内容结构，低频状态再把内容嵌入更长程的语境、任务与交互背景中。这个过程的目标是得到对任务有用、对局部扰动鲁棒、能够跨说话人和跨场景泛化的高层表示。

生成则恰好相反。它更像一个自上而下的展开过程：低频状态先决定这轮话语的意图、风格、说话人一致性和长程上下文；中频状态再把这些高层约束落实到短语级和词级内容组织；高频状态最终把这些约束展开为可听见的韵律、时长、共振峰动态、局部频谱变化和细粒度波形结构。这个过程的目标不是保持不变性，而是生成受控变化。

这意味着真正统一的语音系统，不是让一个表示同时既“尽量抽象”又“尽量细致”，而是允许不同层次的状态承担不同责任。理解和生成可以在高层语义和长程记忆上共享状态，但不必共享全部中低层路径。换句话说，统一不等于对称；统一更像是**共享上层世界模型，分化下层实现路径**。

## 三、文本模态在未来语音模型里更适合扮演“低频锚点”，而不是唯一中枢

从今天的主流路线看，text LLM 往往是整个系统的中枢：speech 先被转换为可接入文本 LM 的表示，模型内部的 reasoning、instruction following 和 context management 主要依赖 text LLM 来完成。这条路之所以成功，是因为文本模型已经非常成熟，而且文本承载高层语义和世界知识的能力确实很强。

但如果我们沿着多时间尺度连续系统继续往前想，文本在语音模型中的角色也许应该被重新理解。文本更适合作为一种**低频语义锚点**：它提供稳定、离散、可组合的高层内容结构，适合承担概念、命题、工具调用、外部知识对齐和显式推理等任务；而 speech 则提供高频、情境化、交互性更强的时间结构，适合承担声学细节、说话风格、话轮组织和在场性。

这启发我们去设计一种真正的多模态分频系统：自然语音 token 或连续语音状态主要落在高频与中频层，负责声学和局部内容；文本 token 则主要落在低频层，负责语义约束与推理控制。两者不需要总是被拼接成同一串序列，更不必强行共享完全同构的输入输出机制。它们可以在较慢层面完成对齐，在较快层面保持各自的动态优势。

这种设计有几个潜在好处。第一，它减少了“为了迁就 text LLM 而过度压缩 speech”的压力。第二，它让 speech 保留自身作为连续交互信号的地位，而不是被完全退化成 text 的附庸。第三，它天然解释了为什么理解和生成可能各自依赖不同 token/状态路径：文本负责低频语义稳定性，语音负责高频生成自然度。

## 四、真正适合语音的骨架，可能是一种“多时间尺度编码—记忆—生成”系统

基于上述判断，我们可以更具体地勾画一种未来语音基础模型的雏形。它不一定只是一种 architecture，也可以是一套建模原则。

第一部分是**多时间尺度编码器**。它不把原始语音一次性压成单一 token，而是同时维护多个时间分辨率的内部状态。高频编码分支负责局部声学与韵律；中频编码分支逐步提取发音片段、词或短语级内容；低频编码分支则整合说话人状态、对话轮次、语用意图和更长程上下文。不同分支之间不是简单拼接，而是通过跨尺度聚合与门控交换信息。

第二部分是**分层记忆系统**。对语音来说，记忆不应只是“更长的 context window”。高频记忆更像 working memory，用于维持最近若干百毫秒到几秒的声学与话轮状态；中频记忆更像 discourse buffer，负责当前句子、短语和局部上下文的连贯性；低频记忆则用于维持说话人风格、角色设定、任务意图、长期对话历史甚至个性化偏好。不同频率的记忆有不同的更新策略：高频记忆更新最勤、容易遗忘；低频记忆更新更慢，但更稳定。这样，模型就不必把所有长期能力都寄托在一个不断增长的 token context 上。

第三部分是**非对称生成器**。理解端主要沿着自下而上的压缩路径工作，而生成端则从低频意图与语义状态出发，逐层向高频展开。这里，中频层可以决定 lexical/prosodic planning，高频层再完成 acoustic rendering。这样，生成就不再只是“把一个文本 token 接一个文本 token 地读出来”，而是真正把上层交互意图动态地投射到连续语音上。

第四部分是**跨模态语义锚定层**。文本、符号动作、工具接口甚至视觉上下文，都可以在这一层作为低频控制信号接入系统。它们不需要吞并 speech，而是与 speech 在较高层次上达成语义对齐，使系统既有 speech-native 的建模能力，又保留 text-native 的推理和显式控制能力。

## 五、这种方向为什么值得做，尽管第一版可能非常难

从工程现实看，上述方向当然更难。它不像“把语音接进现有 LLM 再做 LoRA 微调”那样可以快速得到可工作的系统；它需要重新设计状态更新、跨尺度交互、训练目标、数据配比和评测方式，也很可能在第一版就遇到稳定性、算力和数据组织上的一系列挑战。尤其在现阶段，token 化路线已经非常成熟、社区工具链也围绕它展开，任何偏离这一主流的设计都会面临更高的实验成本。

但正因为难，它才更像真正的下一步。因为现有综述其实已经把主要矛盾说得很清楚：不同 component choice 的作用机制还不够清楚，端到端联合优化仍不足，speech-text 的最佳结合方式不明确，多个预训练组件和不同 frame rate 带来了系统效率与结构上的内在负担。[Cui et al., 2025; Arora et al., 2025] 如果我们只在现有 token 管线里做局部调参，这些问题或许会被暂时缓解，却很难被根本改写。

从研究价值上看，多时间尺度连续系统还有一个重要吸引力：它给语音模型提供了一个真正“语音原生”的理论坐标系。此前很多统一语音模型的创新，都可以被重新解释为对这一坐标系的局部逼近：hierarchical token generation 是粗略的多尺度；dual-token input/output 是理解与生成不对称性的显性表达；streaming/full duplex 是对连续交互时钟的建模；continuous representation 则是对“speech 不必天然离散化”的恢复。也就是说，这并不是凭空发明一个全新方向，而是在已有进展之上，把分散的工程现象上升为统一的系统思想。

## 六、如何从现实可做的工作，逐步走向这个更大的方向

当然，从研究策略看，也没有必要一开始就直接实现完整的多时间尺度连续 speech foundation model。更现实的路径，可能是“先沿着现有架构做结构性逼近，再逐步替换骨架”。

一个可行的第一步，是在现有 text-LLM-based speech model 上显式引入**分频率更新**与**异构输入输出路径**。例如，让输入侧主要接入更偏语义的 speech representation，让输出侧显式使用 acoustic token 或连续 acoustic state；再进一步，在中间层引入不同时间分辨率的 latent state，让模型在高频层只处理局部语音动态，在低频层与 text 语义进行交互。这样的方案仍然可以复用现有 LLM 与 tokenizer/vocoder 生态，但它已经不再把“所有信息都压在同一 token 轴上”视为理所当然。

第二步，可以探索**连续状态替代部分离散 token**。比如，不再要求所有中间层都输出离散 ID，而让高频层保持连续 latent trajectory，仅在较慢层与 text 或语义单位对齐。这样做的目标不是完全抛弃 token，而是把 token 从“唯一统一单位”降级为“某些层次上的对齐与接口单位”。

第三步，才是真正走向**speech-native 的统一骨架**：让模型内部的主要状态组织原则从 token sequence 改为 multi-timescale continuous state evolution。在这一阶段，speech 和 text 不再是简单拼接的两种序列，而是不同频率、不同离散化程度、不同控制职责的耦合信号。

## 七、结语：真正的统一，不是把语音强行写成文本，而是让模型学会按语音的方式组织时间

本章的综述告诉我们，统一语音模型已经走过了非常重要的一步：它不再满足于把识别、理解和生成拆开，而是在端到端建模和多模态交互上迅速推进。与此同时，最新研究也在不断提醒我们：统一模型内部的真正矛盾，并不只是“语音怎么接入大模型”，而是“语音中的不同信息层是否应该共用同一种建模骨架”。

如果把这个问题说得更直接一些，那就是：**语音基础模型的未来，未必是一个更大的文本模型加上一层 speech token；更可能是一个以时间尺度和状态分工为核心组织原则的连续系统。** 在这个系统里，高频层面保留 speech 的声学生命力，中频层面组织内容与短程结构，低频层面承载语义、风格和交互意图；理解是向上压缩，生成是向下展开；文本不再是唯一中枢，而是一个极其重要的低频语义锚点。

这条路当然更难，但也更值得。因为只有当模型真正按语音的方式组织时间，语音基础模型才有可能不再只是“会处理语音的语言模型”，而成为真正意义上的**语音原生基础模型**。
